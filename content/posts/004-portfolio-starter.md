## 1) 先破除迷思：作品集不是「專案數量比賽」

新手最常誤解作品集要很多專案。  
但面試官或合作方其實想看的是：

- 你有沒有把事情做完
- 你怎麼思考問題
- 你遇到問題怎麼解
- 你能不能把結果講清楚

所以：**一個能跑的專案**，比十個半成品更有力。

---

## 2) 最推薦的作品集結構：1 個專案 + 3 個頁面

### Page 1：這個專案解決什麼問題？
用人話寫：

- 目標是什麼？
- 使用者（或你自己）到底痛在哪？
- 你做這個專案的範圍是什麼？

✅ 範例句：  
「我做了一個簡單的待辦清單 API，目標是練習從資料庫到 REST API 的完整流程。」

---

### Page 2：你怎麼做？（把流程拆成 5~8 點）
你不用寫論文，只要能讓別人快速理解：

- 技術選型（為什麼用 Spring Boot？）
- 專案結構（Controller / Service / Repository）
- 主要功能（CRUD、驗證、錯誤處理）
- 測試方式（Postman、單元測試）

---

### Page 3：成果與證據（最重要）
你要放「可被驗證」的東西：

- API 的 demo（GIF 或截圖）
- Postman collection（可下載）
- 部署連結（如果有）
- README（如何跑起來）

---

## 3) 新手做作品集的最小專案題材（3 選 1）

- 待辦清單（Todo）
- 記帳（Expense）
- 書籍/文章收藏（Bookmark）

這三個都有：
- CRUD
- 列表查詢
- 基本驗證

而且不會太大。

---

## 4) 最關鍵的一點：把「踩坑」寫成亮點

你可以在 README 或文章寫：

- 我遇到什麼錯誤（例如 port 被占用、Maven 依賴衝突）
- 我怎麼查
- 我最後怎麼解

這會讓你看起來像：
「做過專案的人」，而不是「只照教學做」。

---

## 重點整理

- 作品集不是比數量：一個能跑的專案就很強
- 用 3 頁講清楚：問題 → 做法 → 成果（證據）
- 把踩坑寫出來，會變成你的亮點
